## Voice and perspective

**Use third person throughout.** The document is a technical analysis, not a personal recommendation.

| Do                             | Don't                  |
| ------------------------------ | ---------------------- |
| "The evaluation found..."      | "We found..."          |
| "The organization requires..." | "Our team requires..." |
| "This document recommends..."  | "I recommend..."       |

**Use active voice.** Passive voice obscures responsibility and reads as evasive.

| Do                                     | Don't                                        |
| -------------------------------------- | -------------------------------------------- |
| "The agent generates specifications"   | "Specifications are generated by the agent"  |
| "CodeRabbit processes code externally" | "Code is processed externally by CodeRabbit" |
| "The pipeline enforces constraints"    | "Constraints are enforced by the pipeline"   |

**State conclusions directly.** Hedging undermines credibility.

| Do                          | Don't                                                      |
| --------------------------- | ---------------------------------------------------------- |
| "Do not adopt Solution A."  | "It may be advisable to consider not adopting Solution A." |
| "The gap is structural."    | "The gap appears to potentially be structural."            |
| "Solution B scores higher." | "Solution B seems to score somewhat higher."               |

## Headings

**Use sentence case.** Capitalize only the first word and proper nouns.

| Do                        | Don't                     |
| ------------------------- | ------------------------- |
| "Evaluation framework"    | "Evaluation Framework"    |
| "Total cost of ownership" | "Total Cost of Ownership" |
| "Risk assessment"         | "Risk Assessment"         |

**Keep headings short and descriptive.** The heading tells readers what they'll find, nothing more.

| Do              | Don't                                                                              |
| --------------- | ---------------------------------------------------------------------------------- |
| "Cost analysis" | "A comprehensive analysis of the total cost of ownership over a three-year period" |
| "Gap analysis"  | "Understanding the gaps between the two solutions"                                 |

## Quantification

**Quantify everything that can be quantified.** Numbers are more precise than adjectives.

| Do                              | Don't                          |
| ------------------------------- | ------------------------------ |
| "42x cost differential"         | "significantly more expensive" |
| "4.45/5.00 vs 2.05/5.00"        | "scored much higher"           |
| "6 risks including 2 High/High" | "more risky"                   |
| "5-10 minutes per plan"         | "takes a while"                |
| "25% weight"                    | "weighted heavily"             |

**Show calculations.** Don't state results without showing how you got there.

```markdown
| Solution A | (1 x 0.25) + (1 x 0.20) + (2 x 0.15) + ... | **2.05** |
```

**Use consistent precision.** If one number has two decimal places, all comparable numbers should have two decimal places.

## Tables vs prose

**Use tables for structured comparisons.** Tables make patterns visible and enable scanning.

Use tables for:

- Feature comparisons
- Scoring matrices
- Cost breakdowns
- Risk matrices
- Capability inventories

Use prose for:

- Analysis and reasoning
- Narrative context
- Explaining methodology
- Drawing conclusions

**Tables must have consistent structure.** Every row follows the same format.

**Headers must be descriptive.** "Solution A" is better than "A".

## Citations and sources

**Cite inline where relevant.** Don't force readers to hunt for the source.

```markdown
Research supports the separated approach. "Large language models cannot
self-correct reasoning yet" (Huang et al., ICLR 2024) demonstrates that...
```

**Collect all sources in Appendix A.** Full links and context for each source.

**Academic format:**

```markdown
- Huang et al., "Title," Venue Year. [arXiv:XXXX.XXXXX](https://arxiv.org/abs/XXXX.XXXXX)
  -- Cited in Sections X, Y. [One sentence explaining relevance.]
```

**Product documentation format:**

```markdown
- [Human-readable label](https://url)
```

## Banned words and patterns

### AI-marker words

These words signal AI-generated text. Replace or remove them.

| Banned word  | Replacement                               |
| ------------ | ----------------------------------------- |
| delve        | examine, analyze, explore                 |
| leverage     | use, apply                                |
| robust       | reliable, stable, strong                  |
| seamless     | smooth, integrated, automatic             |
| utilize      | use                                       |
| tapestry     | (delete the metaphor)                     |
| pivotal      | important, key, critical                  |
| nuanced      | subtle, complex                           |
| multifaceted | complex, varied                           |
| realm        | area, domain, field                       |
| cutting-edge | modern, current, new                      |
| synergy      | (rewrite the sentence)                    |
| holistic     | complete, comprehensive                   |
| paradigm     | approach, model                           |
| innovative   | (remove or state the specific innovation) |
| ecosystem    | environment, system (use sparingly)       |

### Hedging phrases

Delete these. They add no information to an evaluation document.

- "It's important to note that..."
- "It should be considered that..."
- "It bears mentioning that..."
- "One might argue that..."
- "To a certain extent..."
- "In many ways..."
- "Interestingly enough..."
- "It may be advisable to..."

### Filler words

Delete these. The sentence is stronger without them.

- "actually", "basically", "essentially"
- "really", "just", "very", "quite", "rather"

### Em-dash abuse

Em-dashes create parenthetical asides. AI models overuse them. Maximum one pair per paragraph, and prefer commas or parentheses instead.

| Do                                                      | Don't                                                        |
| ------------------------------------------------------- | ------------------------------------------------------------ |
| "The system, which handles authentication, is fast."    | "The system--which handles authentication--is fast."         |
| "The API now supports web. It was designed for mobile." | "The API--originally designed for mobile--now supports web." |

### Journey metaphors

These read as filler. Delete and state the point directly.

| Delete                                | Replace with                        |
| ------------------------------------- | ----------------------------------- |
| "navigate the landscape of options"   | "evaluate the options"              |
| "embark on a journey toward adoption" | "adopt"                             |
| "at the end of the day"               | (delete)                            |
| "moving forward"                      | (delete or state what happens next) |

### Symmetrical lists

AI tends to produce lists where every item follows identical structure. Vary phrasing.

| Don't                                                               | Do                                                |
| ------------------------------------------------------------------- | ------------------------------------------------- |
| "Feature A provides X. Feature B provides Y. Feature C provides Z." | "Feature A handles X. B addresses Y. C solves Z." |

## Diagrams

**Use Mermaid for all architecture diagrams.** Mermaid is version-controllable and renders in most markdown viewers.

**Color code consistently:**

- Red (`fill:#f4cccc,stroke:#cc0000`): Problems, risks, manual steps, gaps, anti-patterns
- Green (`fill:#d9ead3,stroke:#38761d`): Advantages, automation, knowledge sources, good properties

**Label connections** when the relationship is not self-evident:

```mermaid
A -- "manual copy/paste" --> B
```

**Use subgraphs** to group related elements:

```mermaid
subgraph knowledge["Project Knowledge"]
    direction LR
    F["AGENTS.md"]
    G["Rules"]
end
```

**Keep diagrams focused.** One concept per diagram. Two clear diagrams communicate more than one crowded diagram.

## Document metadata

**Status field options:** Draft, Review, Final

**Classification options:** Internal research, Public, Confidential

**Date format:** YYYY-MM-DD (ISO 8601)

**Authors format:** Team or individual name + "under human supervision" when AI-assisted

## Structural consistency

**Solution profiles must be symmetric.** Every field present for Solution A must be present for Solution B. Symmetry enables fair comparison.

**Evaluation criteria weights must sum to 100%.** Verify the arithmetic before publishing.

**Gap analysis must be bidirectional.** Show what A has that B lacks AND what B has that A lacks. One-sided analysis is advocacy.

**Recommendations must include reversal conditions.** This demonstrates intellectual honesty and creates clear re-evaluation triggers.

## Length guidelines

| Section                  | Target length       |
| ------------------------ | ------------------- |
| Executive summary        | 1 page / ~300 words |
| Scope and methodology    | 1-2 pages           |
| Solution profiles (each) | 0.5-1 page          |
| Evaluation framework     | 2-3 pages           |
| Architectural analysis   | 1-2 pages           |
| Cost analysis            | 0.5-1 page          |
| Risk assessment          | 1 page              |
| Gap analysis             | 1 page              |
| Improvement candidates   | 0.5-1 page          |
| Recommendation           | 0.5-1 page          |
| **Total document**       | **10-15 pages**     |

These are guidelines, not hard constraints. Complex evaluations may require more depth. Simple comparisons may need less.

## Common mistakes

**Mistake: Comparing features instead of outcomes.**

- Wrong: "Solution A has feature X. Solution B doesn't."
- Right: "Solution A achieves outcome Y through feature X. Solution B cannot achieve outcome Y."

**Mistake: Scoring without evidence.**

- Wrong: "Pipeline integration: 5"
- Right: "Pipeline integration: 5. Plan feeds directly into the next agent via defined handoff protocol."

**Mistake: Vague mitigations.**

- Wrong: "Mitigation: address the issue"
- Right: "Mitigation: build a translation agent to convert output format (~$10k one-time)"

**Mistake: Missing sensitivity analysis.**

- Always ask: "If I remove the highest-weighted criteria, does the winner change?"
- If yes: the decision depends on that criterion. Justify the weight carefully.
- If no: the decision is robust. State this explicitly.

**Mistake: Orphan recommendations.**

- Every recommendation needs reversal conditions
- Every recommendation needs a re-evaluation date
- Without these, the recommendation cannot adapt to changing circumstances
